# Day 6: A2A - Agent-to-Agent Protocol

> **Week 1: Foundations & First Agents**

## Overview
Today you'll learn Google's Agent-to-Agent (A2A) protocol, which enables independent agents to discover each other, communicate, and delegate tasks. You'll build two agents -- a research agent and a writing agent -- that find each other via Agent Cards and collaborate to complete a task neither could handle alone. You'll also understand how A2A complements MCP rather than competing with it.

## Learning Objectives
- [ ] Understand the A2A protocol: Agent Cards, task lifecycle, and message format
- [ ] Create an Agent Card JSON that advertises an agent's capabilities
- [ ] Implement an A2A server that handles task requests and returns results
- [ ] Build a client agent that discovers another agent and delegates work via A2A
- [ ] Compare MCP (tools for models) vs A2A (communication between agents)

## What You'll Build
Two independent agents -- a Research Agent and a Writing Agent -- that discover each other via Agent Cards, delegate tasks, and collaborate over the A2A protocol to research a topic and produce a summary document.

## Prerequisites
- Completed Days 1-5
- `pip install fastapi uvicorn httpx openai`
- OpenAI API key

## Steps

### Step 1: Understand A2A Architecture
Study the A2A protocol components: **Agent Cards** (JSON at `/.well-known/agent.json` advertising capabilities, skills, and endpoint), **Tasks** (the unit of work with states: submitted, working, input-required, completed, failed), **Messages** (user or agent role with parts: text, file, data), and **Artifacts** (output objects generated by agents). Map out how two agents discover and interact.

### Step 2: Create Agent Cards
Define two Agent Card JSON files. The Research Agent card: `{name: "ResearchAgent", description: "Researches topics using web search", url: "http://localhost:8001", skills: [{id: "research", name: "Topic Research", description: "Searches and synthesizes information on any topic"}]}`. The Writing Agent card: similar structure on port 8002 with a "write_summary" skill. Include the required fields: `apiVersion`, `authentication`, `capabilities`.

### Step 3: Implement the Research Agent A2A Server
Build a FastAPI server that: (1) serves the Agent Card at `GET /.well-known/agent.json`, (2) handles `POST /tasks/send` -- receives a task with a research topic, uses OpenAI + web search to gather information, returns the task with status "completed" and an artifact containing the research results. Implement proper task state transitions and error handling.

### Step 4: Implement the Writing Agent A2A Server
Build a second FastAPI server (port 8002) that: (1) serves its Agent Card, (2) handles `POST /tasks/send` -- receives a task with raw research content, uses OpenAI to write a polished summary, returns the completed task with the summary as an artifact. This agent has no search capability -- it depends on receiving research from another agent.

### Step 5: Build the Orchestrator Client
Create a client agent that: (1) discovers available agents by fetching Agent Cards from known URLs, (2) parses skills to determine which agent can handle which sub-task, (3) sends a research task to the Research Agent, (4) takes the research artifact and sends it to the Writing Agent, (5) returns the final polished summary to the user. This demonstrates the full A2A delegation flow.

### Step 6: Test and Compare with MCP
Run both servers and the orchestrator. Test with: "Research the current state of quantum computing and write a 500-word summary." Verify the task flows through both agents correctly. Then write a comparison: MCP connects models to tools/data (vertical integration); A2A connects agents to agents (horizontal integration). They're complementary -- an agent might use MCP internally and A2A externally.

## Key Concepts
- **Agent Cards:** JSON documents that advertise an agent's identity, capabilities, skills, and endpoint URL. They enable dynamic discovery -- agents can find and evaluate potential collaborators without prior configuration.
- **Task Lifecycle:** Tasks flow through states: submitted -> working -> completed/failed (or input-required for human-in-the-loop). This state machine enables async, long-running agent collaborations.
- **A2A Streaming:** For long tasks, A2A supports Server-Sent Events (SSE) streaming via `tasks/sendSubscribe`, allowing the client to receive incremental updates as the agent works.
- **MCP vs A2A:** MCP is for connecting a model to tools and data sources (like USB for peripherals). A2A is for connecting independent agents to each other (like HTTP for web services). Use MCP inside agents, A2A between agents.
